# 需求变更：AI 提示词优化与参数传递修复

## 变更原因

两处 AI 提示词存在优化空间：
1. **推荐 prompt**：天气城市传的是城市代码（如 `101220101`），非可读城市名；菜品列表仅展示菜名，AI 缺少元数据支撑，匹配不够精准
2. **发现 prompt**：缺少口味偏好、饮食类型、忌口食材等画像信息，AI 生成菜品时无法避开用户禁忌

## 变更内容

### 修改文件

| 文件 | 变更 |
|------|------|
| `weather.py` | `get_weather()` 返回三元组 `(condition, temp, city_name)`，原为二元组 |
| `recommender.py` | `recommend()` / `discover_dishes()` 新增 `weather_city` 参数；两处 prompt 构建函数传城市名而非城市代码 |
| `main.py` | 解包 `get_weather()` 三元组，传递 `weather_city` 给 `recommend()` |
| `update_library.py` | 同 main.py，适配三元组，传递 `weather_city` 给 `discover_dishes()` |
| `tests/test_weather.py` | 测试断言适配三元组返回值 |

## Prompt 变化

### 推荐 prompt（_build_prompt）

改前：菜品仅列菜名
```
已有菜品库（已根据饮食偏好筛选）：麻婆豆腐、番茄炒蛋、...
```

改后：每道菜品附带元数据
```
已有菜品库（已根据饮食偏好筛选，附元数据）：
  - 麻婆豆腐（川菜 | 普通 | 15分钟 | 辣、麻、鲜 | 含辣）
  - 番茄炒蛋（家常菜 | 新手 | 10分钟 | 酸甜、鲜 | 无）
```

### 发现 prompt（_build_discover_prompt）

改前：仅含菜系偏好
```
用户菜系偏好：川菜、徽菜
已有菜品库：...
```

改后：加入口味、饮食类型、忌口信息
```
用户菜系偏好：川菜、徽菜
用户口味偏好：偏咸、偏辣
用户饮食类型：无限制
忌口食材：羊肉、内脏
已有菜品库：...
```

### 日志输出

两处 prompt 发送前均打印完整内容到控制台，方便调试。

### 菜名真实性约束（补充修改）

发现 prompt 新增第 7 条要求，防止 AI 编造菜名：

```
7. **菜名必须是真实存在的菜品名称**，不能自行组合或编造菜名（例如不能将"毛峰茶"和"口水鸡"组合成"毛峰茶香口水鸡"，应直接使用"口水鸡"原名）
```

改前 AI 生成了"春季毛峰茶香口水鸡""徽派笋尖烧臭鳜鱼"等编造菜名；改后全部返回真实菜品名称。

## 菜品库

清空后分两轮填充，共 14 道菜：

| 轮次 | 新增菜品 | 来源 |
|------|----------|------|
| 第1轮 | 口水鸡、臭鳜鱼、毛血旺、刀板香、辣子鸡、胡适一品锅、麻婆豆腐 | 发现 prompt（含菜名约束） |
| 第2轮 | 红烧春笋、川味凉拌折耳根、黄山炖鸽、水煮鱼片、虎皮青椒、绩溪炒粉丝、酸菜鱼 | 发现 prompt（避开已有菜） |

川菜 7 道 + 徽菜 7 道，覆盖新手/普通/熟练三个难度等级，烹饪时间 10~90 分钟。

## 影响范围

- 所有调用链传递城市名而非城市代码，AI 地理位置理解更准确
- 推荐 prompt 增加元数据后 token 消耗略有上升
- `get_weather()` 返回值由 2 变 3，调用方需同步解包
- 发现 prompt 增加菜名真实性约束，AI 不再编造菜名
- 无配置/数据格式变更
