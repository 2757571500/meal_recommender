# 需求分析

## 项目背景

解决"每天吃什么"的决策疲劳。用户每天需要决定午餐和晚餐吃什么，希望结合时令、天气和地区饮食习惯，获得合理的菜品推荐，避免重复。

## 用户目标

1. 每天自动收到贴合时令和天气的三餐推荐
2. 推荐菜品来自已有菜品库，N 天内不重复
3. 菜品库可持续扩充，保持新鲜感
4. 推荐失败时能及时感知

## 功能需求

### 核心功能：每日三餐推荐

| 需求 | 说明 |
|------|------|
| 天气感知 | 根据当天天气状况和温度调整推荐方向 |
| 时令匹配 | 推荐符合当前季节的菜品 |
| 菜品库约束 | 仅从已有菜品库中选择，不自行添加 |
| 短期不重复 | N 天内推荐过的菜品自动排除（默认 7 天） |
| 格式要求 | 区分午餐和晚餐，每餐两菜一汤，附推荐理由 |
| 自动推送 | 结果通过微信推送 |

### 辅助功能：菜品库扩充

| 需求 | 说明 |
|------|------|
| AI 发现新菜 | 根据时令、天气、地区推荐 5-8 道不在库中的新菜 |
| 自动入库 | 新菜自动追加到菜品库，已有菜品更新推荐理由 |
| 独立调度 | 与推荐分开，按周执行 |

### 异常处理

| 场景 | 行为 |
|------|------|
| 天气 API 失败 | 降级为"未知"，继续推荐 |
| AI API 调用失败 | 终止流程，推送失败通知 |
| 推送接口失败 | 打印日志，不阻塞主流程 |
| 菜品库为空 | prompt 中提示"暂无菜品库" |

## 非功能需求

- **无人值守**：青龙面板定时触发，无需人工干预
- **零配置部署**：仅需修改 config.json 即可切换城市、模型、推送地址
- **轻量依赖**：仅依赖 `requests` 库
- **可观测**：失败必须推送通知，不能静默失败
